# 校园二手交易平台 - 视频展示脚本

> **预计时长**：8-10 分钟  
> **制作提示**：每个章节配合屏幕录制 + 旁白/字幕

## 第一章：项目介绍与技术架构（约 1 分 30 秒）

### 1.1 开场（30秒）

**画面**：项目首页截图 + 项目标题动画

**旁白**：

> 大家好，今天我为大家展示的是我开发的**校园二手交易平台**。
>
> 这是一个面向高校学生的闲置物品交易系统，用户可以在平台上发布自己的闲置商品，也可以浏览和购买其他同学发布的商品。
>
> 整个项目采用**前后端不分离**的传统 Web 架构，使用 **JSP 作为视图层**进行服务端渲染，这种架构虽然相对传统，但在学习 SSM 框架的核心原理方面具有很高的教学价值。

---

### 1.2 技术栈详解（1分钟）

**画面**：技术架构图（建议用 PPT 或 draw.io 制作）

**旁白**：

> 下面我来介绍一下项目的技术架构。
>
> **后端框架**方面，我采用的是经典的 **SSM 框架**，也就是 Spring、Spring MVC 和 MyBatis 的组合：
>
> - **Spring 框架**负责整个应用的 IOC 容器管理和 AOP 切面编程，版本是 5.3.21
> - **Spring MVC** 处理 Web 请求的路由分发和数据绑定
> - **MyBatis** 作为持久层框架，负责数据库的 CRUD 操作
>
> **数据库**使用的是 **MySQL 8.0**，搭配阿里巴巴的 **Druid 连接池**进行数据库连接管理。Druid 不仅性能优秀，还自带 SQL 监控功能，后面我会演示这个特性。
>
> **前端**采用 **Bootstrap 4 + jQuery** 的组合，视图层使用 **JSP + JSTL** 标签库进行动态渲染。
>
> 另外，项目还集成了 **PageHelper** 实现 MyBatis 的分页查询，使用 **Lombok** 简化实体类代码，使用 **Commons FileUpload** 处理文件上传。
>
> 整体架构遵循经典的**三层架构**设计：Controller 控制层、Service 业务层、DAO 数据访问层，层次清晰，职责分明。

## 第二章：核心功能演示（约 4 分钟）

### 2.1 用户认证模块（1分钟）

#### 2.1.1 用户注册

**画面**：打开注册页面，填写表单

**旁白**：

> 首先演示用户认证模块。打开注册页面，可以看到我设计了一个简洁美观的注册表单。
>
> 表单中**用户名是登录凭证，必须唯一**；密码部分我做了两个交互优化：
>
> 第一，密码输入框右侧有一个**眼睛图标**，点击可以切换密码的显示和隐藏状态，方便用户确认输入是否正确；
>
> 第二，我增加了**确认密码**输入框，并且实现了**实时校验**功能——当用户输入确认密码时，系统会立即检测两次密码是否一致，并给出绿色或红色的提示。这种即时反馈可以显著提升用户体验。
>
> 昵称字段是可选的，如果用户不填写，系统会自动使用用户名作为昵称。

**操作**：演示密码显示/隐藏、确认密码校验、提交注册

#### 2.1.2 用户登录与验证码

**画面**：打开登录页面

**旁白**：

> 接下来看登录功能。这里我实现了一个重要的**安全特性——图形验证码**。
>
> 验证码是我用 **Java 2D 绘图 API** 动态生成的，每次请求都会生成一张新的验证码图片。
>
> 为了增加安全性，验证码包含了多种干扰元素：**随机颜色的字符**、**干扰线条**、**背景噪点**，而且每个字符都有**随机的旋转角度**，这样可以有效防止 OCR 自动识别。
>
> 点击验证码图片可以刷新获取新的验证码。验证码存储在 Session 中，**有效期为 5 分钟**，并且是**一次性使用**的——验证成功后会立即失效。
>
> 这个设计可以有效**防止暴力破解**和**机器人攻击**，是企业级应用中常见的安全措施。

**操作**：点击刷新验证码，输入错误验证码展示错误提示，输入正确信息登录成功

>

>
> 搜索功能采用的是**模糊匹配**，会同时搜索商品名称和描述内容。后端使用 MyBatis 的动态 SQL 实现条件拼接，配合 **PageHelper 插件**实现分页查询，每页显示固定数量的商品。

#### 2.2.2 商品详情与推荐算法

**画面**：点击商品进入详情页

**旁白**：

> 点击任意商品进入详情页。这里除了展示商品的基本信息、卖家联系方式之外，还有一个我自己实现的**推荐算法模块**。
>
> 页面底部的"相似推荐"区域展示的是**与当前商品同分类的其他热门商品**。这是一种**基于内容的协同过滤**思想：
>
> 算法首先获取当前商品的分类 ID，然后查询同分类下的所有在售商品，按照**浏览量降序排序**，最后取前几个作为推荐结果。这样可以向用户推荐该分类下最受欢迎的商品。
>
> 另外，系统还会**记录用户的浏览历史**。我使用 **ConcurrentHashMap** 存储每个用户的浏览记录，采用 **LRU 策略**限制最多保存 50 条历史，既保证了**线程安全**，又控制了**内存占用**。

**操作**：滚动到页面底部展示推荐区域

#### 2.2.3 发布商品

**画面**：进入发布商品页面

**旁白**：

> 接下来演示发布商品功能。填写商品名称、价格、选择分类、编写描述，然后上传商品图片。
>
> 图片上传功能使用的是 **Apache Commons FileUpload** 组件。为了防止文件名冲突，我使用 **UUID 生成唯一的文件名**。
>
> 这里采用的是**文件存储分离**策略：图片文件保存在服务器的文件系统中，数据库只存储图片的**相对路径**。这种设计有三个好处：
>
> 第一，减少数据库存储压力，避免存储大量二进制数据；
> 第二，图片可以通过静态资源方式直接访问，性能更好；
> 第三，便于后期迁移到 CDN 或对象存储服务。

**操作**：填写表单、上传图片、提交发布

### 2.3 订单模块（1分钟）

#### 2.3.1 下单购买

**画面**：在商品详情页点击购买

**旁白**：

> 现在演示购买流程。在商品详情页点击"立即购买"按钮，系统会弹出确认对话框。
>
> 确认购买后，系统会创建订单并更新商品状态。这里涉及到一个关键的技术点——**Spring 声明式事务**。
>
> 在 OrderServiceImpl 的 createOrder 方法上，我使用了 `@Transactional(rollbackFor = Exception.class)` 注解。这个注解的作用是：**订单创建**和**商品状态更新**这两个操作会在**同一个事务**中执行。
>
> 如果任意一个操作失败，整个事务会**自动回滚**，保证数据的一致性。这也是**防止超卖**的关键——在创建订单之前，会先检查商品状态是否为"在售"，如果商品已被其他用户购买，事务会抛出异常并回滚。

**操作**：点击购买、确认、展示成功提示

#### 2.3.2 卖家订单管理

**画面**：进入"我的销售"页面

**旁白**：

>
> 每个待处理的订单都有两个操作按钮：**确认完成**和**取消订单**。
>
> 点击"确认完成"，订单状态变为"已完成"，表示交易成功；
>
> 点击"取消订单"，系统不仅会把订单状态改为"已取消"，还会**自动将商品恢复为在售状态**，这样商品可以重新被其他用户购买。
>
> 取消订单的操作同样使用了**事务管理**，确保订单状态更新和商品状态恢复是**原子操作**。

**操作**：演示确认完成和取消订单

## 第三章：技术创新点深度解析（约 3 分钟）⭐重点⭐

### 3.1 AOP 声明式日志（1分钟）

**画面**：展示代码 - Log.java、LogAspect.java

**旁白**：

> 接下来我要重点介绍项目中的几个技术创新点。
>
> 第一个是**基于 AOP 的声明式日志记录**。这是我觉得最有技术含量的一个功能。
>
> 首先，我定义了一个自定义注解 `@Log`，它只有一个 value 属性用于描述操作内容。然后，我编写了一个切面类 `LogAspect`，使用 `@Aspect` 注解标记。
>
> 在切面中，我定义了一个**切入点** `@Pointcut("@annotation(com.campus.annotation.Log)")`，表示拦截所有被 @Log 注解标记的方法。
>
> 核心逻辑在 `@Around` 环绕通知中实现：
>
> 方法执行前，我记录开始时间；方法执行后，我通过**反射获取注解的值**，然后从 Spring 的 RequestContextHolder 获取当前请求和 Session，从而获取到**操作用户、请求参数、IP 地址**等信息。
>
> 最后计算方法执行耗时，将所有信息格式化输出到日志中。
>
> 这种设计的好处是：开发者只需要在方法上加一个简单的注解，就能自动记录操作日志，**业务代码和日志代码完全解耦**。这就是 AOP 面向切面编程的核心思想。

**画面**：展示控制台日志输出

**旁白**：

> 可以看到控制台输出的日志，包含了用户名、操作描述、方法签名、执行耗时、IP 地址等完整信息。

---

### 3.2 全局异常处理（30秒）

**画面**：展示代码 - GlobalExceptionHandler.java

**旁白**：

> 第二个创新点是**全局异常处理器**。
>
> 我使用 Spring 的 `@ControllerAdvice` 注解创建了一个全局异常处理类。这个类中定义了多个 `@ExceptionHandler` 方法，分别处理不同类型的异常。
>
> 比如 BusinessException 是我自定义的业务异常，会返回具体的错误信息；而其他未知异常会统一返回"系统繁忙"的提示，同时在后台记录详细的错误堆栈。
>
> 所有异常响应都使用统一的 `Result<T>` 格式封装，包含 success、code、message、data、timestamp 五个字段。这样前端可以用**统一的方式处理所有 API 响应**，代码更加规范。
>
> 这个设计**避免了在每个 Controller 中重复编写 try-catch**，大大提高了代码的可维护性。

---

### 3.3 Druid 数据源监控（30秒）

**画面**：访问 /druid/ 监控页面

**旁白**：

> 第三个创新点是集成了 **Druid 数据源监控**。
>
> 访问 `/druid/` 路径，输入配置好的账号密码，可以进入 Druid 的监控面板。
>
> 这个面板提供了非常丰富的监控信息：
>
> - **SQL 监控**：可以看到所有执行过的 SQL 语句、执行次数、最大执行时间、平均执行时间
> - **慢查询日志**：自动统计执行时间超过阈值的 SQL，帮助发现性能瓶颈
> - **连接池状态**：实时查看活跃连接数、等待线程数、连接池配置等
>
> 这是**企业级应用中常用的性能监控方案**，对于排查数据库性能问题非常有帮助。

---

### 3.4 权限拦截器（30秒）

**画面**：展示代码 - LoginInterceptor.java、AdminInterceptor.java

**旁白**：

> 第四个创新点是**多级权限拦截器**。
>
> 我实现了两个拦截器：`LoginInterceptor` 拦截需要登录才能访问的页面；`AdminInterceptor` 拦截只有管理员才能访问的后台页面。
>
> 拦截器的工作原理是：在 `preHandle` 方法中检查 Session 中是否存在用户信息，以及用户的角色是否满足要求。如果验证失败，返回 false 阻止请求继续处理，同时重定向到登录页或首页。
>
> 在 Spring MVC 配置文件中，我通过 `<mvc:interceptors>` 标签配置了拦截规则，指定了哪些路径需要被拦截，哪些路径可以放行。
>
> 这种基于拦截器的权限控制方式，相比在每个 Controller 中手动检查权限，**更加集中、易于维护**。
