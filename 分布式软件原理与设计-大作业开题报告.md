# 分布式软件原理与设计大作业开题报告

## 一、课题名称
基于发布-订阅模式的新商品实时推荐系统——校园二手交易平台智能推送能力建设

## 二、研究背景与问题分析

### 2.1 项目现状
本课题基于一个已实现完整业务闭环的 SSM（Spring + Spring MVC + MyBatis + JSP）校园二手交易平台。该平台具备用户注册登录、商品发布购买、订单管理、后台统计等核心功能。

### 2.2 真实问题：新商品上架后无人问津

在现有系统中，当一个卖家发布一个新商品后，这个商品会出现在商品列表和搜索结果中，但**系统不会主动将它推荐给可能感兴趣的用户**。新商品只能被动等待用户搜索或浏览时被发现。

**真实用户场景**：
- 卖家小王发布了一本《算法导论》，但因为没有浏览量，它在商品列表中排在很后面
- 买家小李是计算机系的学生，他之前浏览过《Java编程思想》《数据结构》等书籍
- 小李打开平台，看到的是热门商品推荐（可能是《考研政治》这类浏览量高的商品）
- 小王的书没人看到，小李找不到想买的书——**供需双方都受损**

**这个问题的本质是什么？**

这是一个典型的**信息分发**问题——卖家发布了新商品（信息生产者），系统需要找到对这个商品感兴趣的用户（信息消费者），并将商品信息推送给这些用户。

在分布式系统中，这个问题对应的是**发布-订阅（Pub-Sub）模式**：生产者发布事件，系统根据订阅规则将事件路由到感兴趣的消费者。

### 2.3 为什么这个问题适合用分布式思路解决？

发布-订阅模式是分布式系统的核心设计模式之一。在分布式消息中间件（如 Kafka、RabbitMQ）中，核心要解决的就是"如何高效地将消息匹配并路由到感兴趣的消费者"。

在我们的场景中：
- **发布者**：卖家（发布新商品）
- **事件**：新商品上架（包含商品特征：分类、价格、关键词等）
- **订阅者**：买家（对某些类型的商品感兴趣）
- **匹配规则**：商品特征与用户兴趣画像的匹配度

当用户量和商品量增长到一定规模时，这个匹配问题会面临分布式系统特有的挑战：
1. **匹配效率**：如何在海量用户中快速找到对某个商品感兴趣的人？
2. **实时性**：商品发布后，推荐结果需要在秒级内生效
3. **可扩展性**：用户量和商品量增长时，匹配系统需要能水平扩展

### 2.4 现有系统的不足

现有系统在解决这个问题上存在三个不足：

**不足一：没有用户兴趣画像**
现有系统虽然记录了用户的浏览历史，但从未对用户的兴趣进行建模。系统不知道"这个用户喜欢什么类型的商品"，只知道"这个用户看过哪些商品"。

**不足二：没有主动推送机制**
现有推荐是被动的——用户访问推荐接口时才计算。新商品上架后，系统不会主动将它推送给潜在感兴趣的用户。

**不足三：推荐计算效率低**
现有个性化推荐方法存在 N+1 查询问题，遍历浏览历史时对每个商品都查一次数据库，响应时间可达数秒。

## 三、研究目标

### 3.1 核心目标
实现一个基于发布-订阅模式的新商品实时推荐系统：当卖家发布新商品时，系统能根据商品特征和用户兴趣画像，实时找到对该商品感兴趣的用户，并将商品推荐给他们。

### 3.2 具体目标
- **用户兴趣画像构建**：基于用户浏览历史、购买记录等行为数据，构建每个用户的兴趣向量
- **商品特征提取**：从商品分类、价格、标题关键词等维度提取商品特征向量
- **实时匹配推荐**：新商品发布后，实时计算与用户兴趣的匹配度，找到 TopN 潜在感兴趣用户
- **分布式缓存加速**：使用 Redis 缓存用户兴趣画像和商品特征，避免每次推荐都查数据库
- **可扩展架构**：匹配系统支持水平扩展，用户量和商品量增长时性能不退化

## 四、技术方案

### 4.1 核心架构：发布-订阅模式

#### 4.1.1 架构设计

```
┌──────────────┐    发布新商品事件    ┌──────────────────┐
│  卖家发布商品  │ ──────────────────→ │  事件匹配引擎     │
└──────────────┘                      └───────┬──────────┘
                                              │
                    ┌─────────────────────────┼──────────┐
                    │                         │          │
                    ▼                         ▼          ▼
            ┌──────────────┐         ┌──────────────┐
            │  用户兴趣画像   │         │  商品特征索引   │
            │  (Redis Hash)  │         │  (Redis)      │
            └──────────────┘         └──────────────┘
                    │                         │
                    └─────────────┬───────────┘
                                  │
                                  ▼
                        ┌──────────────────┐
                        │  匹配结果输出      │
                        │  (TopN 感兴趣用户) │
                        └──────────────────┘
```

#### 4.1.2 核心流程

**第一步：用户兴趣画像构建（离线 + 增量）**

当用户浏览商品时，系统实时更新用户的兴趣画像：

```
用户兴趣画像 = {
    "category_weights": {           // 分类偏好权重
        "闲置书籍": 0.6,
        "数码产品": 0.3,
        "生活用品": 0.1
    },
    "price_range": [0, 200],        // 价格偏好区间
    "keywords": ["Java", "编程"],    // 关键词偏好
    "avg_price": 85,                // 平均浏览价格
    "active_time": "2026-04-28 16:00:00"  // 最近活跃时间
}
```

**第二步：商品特征提取（实时）**

卖家发布新商品时，系统提取商品特征：

```
商品特征 = {
    "category_id": 1,               // 分类ID
    "category_name": "闲置书籍",     // 分类名称
    "price": 45.00,                 // 价格
    "keywords": ["算法", "计算机"],  // 标题关键词
    "seller_id": 123,               // 卖家ID
    "publish_time": "2026-04-28 16:05:00"
}
```

**第三步：匹配计算（实时）**

新商品发布后，系统计算该商品与所有活跃用户的匹配度：

```
match_score(user, product) = 
    w1 × category_match(user, product) +      // 分类匹配度
    w2 × price_match(user, product) +          // 价格匹配度
    w3 × keyword_match(user, product)          // 关键词匹配度
```

匹配度计算公式中的三个维度：

1. **分类匹配度**：用户对该分类的偏好权重（来自用户兴趣画像）
2. **价格匹配度**：商品价格是否在用户常浏览的价格区间内
3. **关键词匹配度**：商品标题关键词与用户浏览历史关键词的相似度

**第四步：结果输出**

取匹配度最高的 TopN 用户，将新商品推荐给他们。

### 4.2 分布式缓存设计

#### 4.2.1 Redis 数据结构设计

| 缓存数据 | 数据结构 | Key 设计 | TTL | 说明 |
|---------|---------|---------|-----|------|
| 用户兴趣画像 | Hash | `user:profile:{userId}` | 24小时 | 增量更新 |
| 商品特征 | Hash | `product:feature:{productId}` | 7天 | 发布时写入 |
| 分类-用户索引 | Set | `idx:category:{categoryId}:users` | 24小时 | 对该分类感兴趣的用户 |
| 关键词-用户索引 | Set | `idx:keyword:{keyword}:users` | 24小时 | 对该关键词感兴趣的用户 |
| 用户推荐结果 | List | `rec:inbox:{userId}` | 6小时 | 推送给用户的商品列表 |

#### 4.2.2 索引加速

为了加速匹配过程，引入两级索引：

**一级索引：分类-用户索引**
- 记录每个分类下有哪些用户感兴趣
- 新商品发布时，直接从这个分类的用户集合中筛选
- 避免遍历所有用户

**二级索引：关键词-用户索引**
- 记录每个关键词下有哪些用户感兴趣
- 在分类筛选的基础上，进一步按关键词精确定位

**匹配流程优化**：
```
新商品发布（分类=闲置书籍，关键词=算法）
    ↓
1. 从 Redis 读取 idx:category:闲置书籍:users → 得到用户集合 U1
2. 从 Redis 读取 idx:keyword:算法:users → 得到用户集合 U2
3. 取 U1 ∩ U2 的交集 → 得到高意向用户集合 U3
4. 对 U3 中的用户计算完整匹配度
5. 取 TopN 输出
```

这样就不需要遍历所有用户，只需要在索引命中的用户集合中计算即可。

### 4.3 与分布式课程内容的关联

本方案与分布式软件原理与设计课程的核心内容紧密关联，涉及以下六个关键知识点：

#### 4.3.1 发布-订阅模式（Pub-Sub）

**课程对应知识点**：分布式消息中间件的核心设计模式，Kafka、RabbitMQ 等消息队列的核心机制。

**方案中的体现**：
- **事件模型**：将"新商品上架"建模为事件（Event），商品特征（分类、价格、关键词）作为事件内容
- **订阅规则**：将"用户兴趣画像"建模为订阅（Subscription），用户对哪些分类、价格区间、关键词感兴趣即为订阅条件
- **事件路由**：匹配引擎负责将新商品事件路由到符合条件的用户，相当于消息中间件中的 Broker 路由功能
- **解耦**：卖家和买家之间通过匹配引擎解耦，卖家不需要知道谁会对他的商品感兴趣，买家也不需要主动搜索

**与课程理论的对应关系**：

| 分布式消息中间件概念 | 本方案中的对应 |
|-------------------|--------------|
| Producer（生产者） | 卖家（发布新商品） |
| Event/Message（事件/消息） | 新商品上架事件（含商品特征） |
| Topic（主题） | 商品分类（如"闲置书籍"是一个主题） |
| Subscription（订阅） | 用户兴趣画像（用户订阅了某些分类/关键词） |
| Consumer（消费者） | 买家（接收推荐商品） |
| Broker（路由器） | 匹配引擎（将事件路由到感兴趣的消费者） |
| Routing Key（路由键） | 商品特征向量（分类ID、关键词等） |
| Binding Key（绑定键） | 用户兴趣画像中的偏好维度 |

**为什么这是分布式系统的核心问题**：在分布式消息中间件中，当有大量主题和大量订阅者时，如何高效地将消息匹配到订阅者是一个经典问题。我们的两级索引方案正是解决这个问题的典型方法——与 Kafka 的分区索引、RabbitMQ 的绑定表路由本质上是同一类思路。

#### 4.3.2 分布式缓存

**课程对应知识点**：分布式缓存（Redis/Memcached）、缓存一致性、缓存穿透/击穿/雪崩。

**方案中的体现**：
- **缓存数据**：用户兴趣画像（Hash）、商品特征（Hash）、两级索引（Set）、推荐结果（List）全部存储在 Redis 中
- **多实例一致性**：当系统部署多个 Web 实例时，所有实例从同一个 Redis 集群读取画像和索引数据，保证数据一致性。如果使用本地内存缓存，不同实例之间数据会不一致
- **缓存策略**：
  - 用户兴趣画像 TTL 24 小时，每次用户浏览行为触发增量更新并刷新 TTL
  - 商品特征 TTL 7 天，商品下架时主动删除
  - 索引数据 TTL 24 小时，定时任务重建
- **缓存穿透防护**：如果 Redis 中查不到用户画像，不会直接返回空，而是从数据库加载历史行为重建画像并回填 Redis

**与课程理论的对应关系**：

| 分布式缓存概念 | 本方案中的对应 |
|--------------|--------------|
| 缓存命中率 | 画像缓存命中率、索引缓存命中率 |
| 缓存一致性 | 用户浏览行为触发增量更新，保证画像与最新行为一致 |
| 缓存穿透 | 画像为空时从数据库重建，不回空 |
| TTL 过期策略 | 画像 24h、商品特征 7天、索引 24h |
| 分布式共享 | 多实例共享同一 Redis 集群，消除本地缓存不一致 |

#### 4.3.3 分布式索引

**课程对应知识点**：分布式系统中的索引结构（倒排索引、二级索引）、空间换时间思想。

**方案中的体现**：
- **为什么需要分布式索引**：如果平台有 10 万用户，每次发布新商品都遍历所有用户计算匹配度，时间复杂度 O(n)，不可接受。需要建立索引将匹配范围缩小到 O(k)，其中 k << n
- **一级索引（分类-用户索引）**：以商品分类为 Key，记录对该分类感兴趣的用户 ID 集合。数据结构为 Redis Set，支持 O(1) 的成员判断和交集运算
- **二级索引（关键词-用户索引）**：以关键词为 Key，记录对该关键词感兴趣的用户 ID 集合。进一步精确定位
- **索引构建**：基于用户兴趣画像中的分类权重和关键词列表，异步构建两级索引
- **索引更新**：用户兴趣变化时（浏览新分类的商品），增量更新索引

**索引加速的效果分析**：

```
无索引：遍历 100,000 用户 × 计算匹配度 = 100,000 次计算
有索引：分类索引命中 5,000 人 ∩ 关键词索引命中 200 人 = 50 人
        只需要计算 50 人的匹配度，效率提升 2000 倍
```

**与课程理论的对应关系**：这本质上是分布式系统中"倒排索引"思想的体现——以内容（分类/关键词）为键，记录包含该内容的文档（用户）列表。Elasticsearch、Solr 等分布式搜索引擎的核心机制与此一致。

#### 4.3.4 离线/在线分离（Lambda 架构）

**课程对应知识点**：Lambda 架构（离线批处理 + 实时流处理）、批流一体。

**方案中的体现**：
- **离线全量计算（Batch Layer）**：每天凌晨通过定时任务，扫描所有用户的完整浏览历史，重新计算用户兴趣画像的全量数据。保证画像的准确性，修正增量更新可能积累的偏差
- **在线增量更新（Speed Layer）**：用户每次浏览商品时，实时更新兴趣画像中的分类权重、关键词列表、价格区间。保证画像的实时性，用户兴趣变化后立即生效
- **合并（Serving Layer）**：推荐时读取的是"离线全量 + 在线增量"合并后的画像数据，既准确又实时

**为什么需要离线/在线分离**：
- 如果只用离线计算：用户兴趣变化后需要等到第二天才能反映到推荐中，实时性差
- 如果只用在线增量：长时间累积可能导致权重偏差（比如用户偶然浏览了一个不感兴趣的商品，权重被错误提高）
- 两者结合：离线全量修正偏差，在线增量保证实时

**与课程理论的对应关系**：这正是 Lambda 架构在推荐系统中的一个典型应用——Batch Layer 保证数据的准确性（Accuracy），Speed Layer 保证数据的实时性（Low Latency），Serving Layer 合并两者结果提供服务。

#### 4.3.5 降级与容错

**课程对应知识点**：分布式系统的容错设计、服务降级、熔断机制、故障隔离。

**方案中的体现**：
- **三级降级策略**：
  - L1（Redis 不可用）→ 数据库直查 + 分类热门推荐
  - L2（用户画像为空）→ 全局热门推荐
  - L3（匹配结果为空）→ 同分类热门推荐
- **故障隔离**：Redis 缓存故障不影响平台的基本功能（商品浏览、搜索、购买等），只影响推荐模块的个性化效果
- **优雅降级**：降级不是直接报错，而是返回次优结果，保证用户体验不中断

**与课程理论的对应关系**：分布式系统中，任何组件都可能故障。容错设计的核心原则是"Design for Failure"——假设故障一定会发生，提前设计好降级路径。我们的三级降级策略正是这一原则的实践。

#### 4.3.6 可观测性

**课程对应知识点**：分布式系统的可观测性（Metrics/Tracing/Logging）、监控告警。

**方案中的体现**：
- **Metrics（指标）**：匹配计算耗时、推荐推送量、推荐点击率、缓存命中率
- **Tracing（链路追踪）**：一次推荐请求的完整链路——商品发布→特征提取→索引查询→匹配计算→结果推送，各环节耗时可追踪
- **Logging（日志）**：每次匹配计算记录详细日志，包括商品信息、命中用户数、TopN 结果、各维度匹配度分布

**与课程理论的对应关系**：可观测性是分布式系统的关键能力。在单机系统中，可以通过单进程调试来排查问题；但在分布式系统中，请求会经过多个节点，必须通过 Metrics/Tracing/Logging 来理解系统行为。


### 4.4 降级与容错

| 故障场景 | 降级策略 |
|---------|---------|
| Redis 不可用 | 降级为数据库直查，按分类筛选热门商品 |
| 用户兴趣画像为空（新用户） | 降级为热门商品推荐 |
| 匹配结果为空 | 降级为同分类热门商品 |
| 索引数据未就绪 | 降级为全量用户遍历（限制用户数量） |

### 4.5 可观测性

| 指标 | 说明 |
|------|------|
| 匹配计算耗时 | 新商品发布后匹配计算的耗时 |
| 推荐推送量 | 每小时推送给用户的商品数量 |
| 推荐点击率 | 用户点击推荐商品的比例 |
| 缓存命中率 | 用户兴趣画像和索引的缓存命中率 |
| 各维度匹配度分布 | 分类/价格/关键词匹配度的统计分布 |

## 五、实施计划

### 第一阶段：用户兴趣画像（第 1-3 周）
- 设计用户兴趣画像的数据结构
- 实现基于浏览历史的兴趣权重计算
- 实现兴趣画像的 Redis 存储与增量更新
- 实现离线全量计算定时任务

### 第二阶段：匹配引擎（第 4-6 周）
- 实现商品特征提取
- 实现匹配度计算（分类/价格/关键词三维度）
- 实现两级索引结构（分类索引 + 关键词索引）
- 实现 TopN 匹配算法

### 第三阶段：治理与观测（第 7-8 周）

## 六、预期成果

### 6.1 工程成果
- 用户兴趣画像系统（基于浏览行为的兴趣建模 + Redis 存储）
- 商品特征提取系统（分类/价格/关键词多维特征）
- 实时匹配引擎（三维度匹配度计算 + 两级索引加速）
- 降级与容错策略（Redis 不可用、画像为空等场景）
- 可观测性指标采集（匹配耗时、推送量、点击率）

### 6.2 可验证的优化效果

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 推荐机制 | 用户主动访问时才推荐 | 新商品发布后主动匹配用户 |
| 用户兴趣建模 | 无（仅记录浏览历史） | 多维兴趣画像（分类/价格/关键词） |
| 新商品曝光 | 被动等待搜索 | 发布后实时推送给潜在买家 |
| 匹配效率 | 遍历所有用户（N+1查询） | 索引加速（毫秒级匹配） |
| 多实例一致性 | 不一致 | 一致（Redis 共享） |

- 监控指标展示

## 七、风险分析

| 风险 | 应对策略 |
|------|---------|
| 用户兴趣画像不准确 | 逐步优化权重计算，引入更多行为信号（浏览时长、购买行为） |
| 匹配计算量过大 | 两级索引加速 + 限制候选用户数量 + 离线预计算 |
| 新用户冷启动 | 降级为热门商品推荐，积累行为数据后逐步个性化 |
| Redis 引入后运维复杂度 | 使用 Docker 部署 Redis，简化运维 |
