# 分布式软件原理与设计大作业开题报告

## 一、课题名称
基于发布-订阅模式的新商品实时推荐系统——校园二手交易平台智能推送能力建设

## 二、研究背景与问题分析

### 2.1 项目现状
本课题基于一个已实现完整业务闭环的 SSM（Spring + Spring MVC + MyBatis + JSP）校园二手交易平台。该平台具备用户注册登录、商品发布购买、订单管理、后台统计等核心功能。

### 2.2 真实问题：新商品上架后无人问津

在现有系统中，当一个卖家发布一个新商品后，这个商品会出现在商品列表和搜索结果中，但**系统不会主动将它推荐给可能感兴趣的用户**。新商品只能被动等待用户搜索或浏览时被发现。

**真实用户场景**：
- 卖家小王发布了一本《算法导论》，但因为没有浏览量，它在商品列表中排在很后面
- 买家小李是计算机系的学生，他之前浏览过《Java编程思想》《数据结构》等书籍
- 小李打开平台，看到的是热门商品推荐（可能是《考研政治》这类浏览量高的商品）
- 小王的书没人看到，小李找不到想买的书——**供需双方都受损**

**这个问题的本质是什么？**

这是一个典型的**信息分发**问题——卖家发布了新商品（信息生产者），系统需要找到对这个商品感兴趣的用户（信息消费者），并将商品信息推送给这些用户。

在分布式系统中，这个问题对应的是**发布-订阅（Pub-Sub）模式**：生产者发布事件，系统根据订阅规则将事件路由到感兴趣的消费者。

### 2.3 为什么这个问题适合用分布式思路解决？

发布-订阅模式是分布式系统的核心设计模式之一。在分布式消息中间件（如 Kafka、RabbitMQ）中，核心要解决的就是"如何高效地将消息匹配并路由到感兴趣的消费者"。

在我们的场景中：
- **发布者**：卖家（发布新商品）
- **事件**：新商品上架（包含商品特征：分类、价格、关键词等）
- **订阅者**：买家（对某些类型的商品感兴趣）
- **匹配规则**：商品特征与用户兴趣画像的匹配度

当用户量和商品量增长到一定规模时，这个匹配问题会面临分布式系统特有的挑战：
1. **匹配效率**：如何在海量用户中快速找到对某个商品感兴趣的人？
2. **实时性**：商品发布后，推荐结果需要在秒级内生效
3. **可扩展性**：用户量和商品量增长时，匹配系统需要能水平扩展

### 2.4 现有系统的全面问题诊断

通过对现有系统全部 34 个 Java 文件的代码审查，我们发现了以下 18 个具体问题，按严重程度分为三类：

#### 🔴 严重问题（直接影响核心功能）

| 编号 | 问题 | 具体表现 | 影响 |
|------|------|---------|------|
| P1 | **浏览历史存储在本地内存** | 使用 `ConcurrentHashMap<Integer, LinkedList<Integer>>` 存储浏览历史 | ① 应用重启后数据全部丢失 ② 多实例部署时各实例数据不共享 ③ 无持久化，用户换设备登录历史消失 |
| P2 | **N+1 查询导致性能低下** | `getPersonalizedRecommendations()` 中遍历浏览历史，对每个商品 ID 都调用 `productMapper.findById()` 查一次数据库 | 浏览 50 个商品就需要查 50 次数据库，响应时间可达数秒 |
| P3 | **推荐是被动触发的** | 推荐只在用户访问 `/product/recommendations` 接口时才计算，新商品发布后没有任何主动推送机制 | 新商品上架后只能被动等待，无法主动触达潜在买家 |
| P4 | **没有用户兴趣画像** | 系统只记录了"用户看过哪些商品"的原始浏览历史，没有对用户的兴趣进行结构化建模 | 无法回答"这个用户喜欢什么类型的商品"这个核心问题 |

#### 🟡 中等问题（影响推荐效果和系统能力）

| 编号 | 问题 | 具体表现 | 影响 |
|------|------|---------|------|
| P5 | **相似商品推荐过于简单** | `getSimilarProducts()` 只是按同分类 + 按浏览量排序，没有真正的匹配度计算 | 推荐结果缺乏个性化，无法区分用户对不同商品的偏好差异 |
| P6 | **没有分布式缓存** | pom.xml 中没有引入 Redis 客户端依赖（jedis/lettuce），所有数据都直接从数据库读取 | ① 高并发下数据库压力大 ② 多实例间无法共享数据 ③ 无法实现缓存加速 |
| P7 | **没有定时任务能力** | pom.xml 中没有引入 Spring Task 或 Quartz 依赖 | 无法实现离线全量计算、索引重建、画像更新等定时任务 |
| P8 | **没有降级与容错机制** | 推荐服务没有熔断/降级设计，Redis 不可用时没有备选方案 | 依赖组件故障时推荐功能完全不可用 |
| P9 | **没有可观测性** | 没有指标采集（Metrics）、链路追踪（Tracing）、结构化日志（Logging） | 无法监控推荐系统的运行状态和性能瓶颈 |

#### 🔵 轻微问题（影响系统完整性和扩展性）

| 编号 | 问题 | 具体表现 | 影响 |
|------|------|---------|------|
| P10 | **用户实体缺少在线状态** | User 实体没有 `lastLoginTime`、`onlineStatus` 等字段 | 无法判断用户是否在线，无法实现"仅推送给在线用户"的优化 |
| P11 | **商品发布后没有事件通知** | `ProductServiceImpl.publish()` 只是插入数据库，没有触发任何后续事件 | 无法在商品发布后自动触发匹配推荐流程 |
| P12 | **数据库没有浏览历史表** | 浏览历史只存在内存中，数据库没有对应的持久化表 | ① 重启丢失 ② 无法做离线全量分析 ③ 无法跨会话追踪 |
| P13 | **没有关键词提取机制** | 商品标题没有做分词处理，无法提取关键词用于匹配 | 关键词匹配维度无法实现，只能依赖分类进行粗粒度推荐 |
| P14 | **Session 存储在本地** | 用户登录状态存储在 Tomcat 本地 Session 中 | 多实例部署时需要引入 Session 共享机制（如 Redis Session） |
| P15 | **管理员统计使用全表查询** | `AdminController.index()` 中 `userService.findAll().size()` 获取用户数，`orderService.findAll().size()` 获取订单数 | 数据量大时全表查询导致页面加载缓慢，应使用 COUNT 聚合查询 |
| P16 | **商品详情页重复查询数据库** | `ProductController.detail()` 中先 `findById()` 获取商品，`increaseViewCount()` 后再次 `findById()` 重新查询 | 每次浏览商品都执行两次数据库查询，增加数据库压力 |
| P17 | **Service 层缺少安全校验** | `OrderServiceImpl.createOrder()` 没有校验买家不能购买自己的商品，该逻辑只在 Controller 层实现 | 绕过 Controller 直接调用 Service 时存在安全隐患 |
| P18 | **登录后未更新在线状态** | `UserServiceImpl.login()` 验证通过后直接返回 User 对象，没有更新 `lastLoginTime` 和 `onlineStatus` | 无法追踪用户活跃状态，影响画像中的活跃度指标计算 |

### 2.5 问题根源分析

上述 18 个问题可以归结为三个根源：

**根源一：架构设计上缺乏分布式思维**
系统采用单机架构设计，所有状态（浏览历史、Session）都存储在本地内存中。这导致系统无法水平扩展，也无法实现多实例间的数据共享。

**根源二：推荐算法过于简单**
现有推荐只是"同分类按浏览量排序"和"基于浏览历史的分类频次统计"，没有真正的匹配度计算，也没有用户兴趣建模。

**根源三：缺少事件驱动机制**
商品发布和用户浏览等行为没有触发后续的异步处理流程，导致新商品无法主动推送给潜在买家。

## 三、研究目标

### 3.1 核心目标
实现一个基于发布-订阅模式的新商品实时推荐系统：当卖家发布新商品时，系统能根据商品特征和用户兴趣画像，实时找到对该商品感兴趣的用户，并将商品推荐给他们。

### 3.2 具体目标

| 目标编号 | 目标描述 | 解决对应问题 |
|---------|---------|------------|
| G1 | **用户兴趣画像构建**：基于用户浏览历史、购买记录等行为数据，构建每个用户的多维兴趣向量（分类偏好权重、价格区间、关键词偏好） | P4（没有用户兴趣画像） |

### 4.2 核心流程

#### 4.2.1 用户兴趣画像构建

**数据结构设计**：

```java
// 用户兴趣画像（Redis Hash + MySQL 持久化）
UserProfile {
    // 分类偏好权重（key: categoryId, value: 归一化权重）
    Map<Integer, Double> categoryWeights;
    
    // 价格偏好区间
    Double minPrice;  // 浏览过的最低价格
    Double maxPrice;  // 浏览过的最高价格
    Double avgPrice;  // 平均浏览价格
    
    // 关键词偏好（key: 关键词, value: 出现频次）
    Map<String, Integer> keywords;
    
    // 活跃度指标
    Date lastBrowseTime;   // 最近浏览时间
    Date lastLoginTime;    // 最近登录时间
    Integer browseCount;   // 总浏览数（近30天）
    Integer purchaseCount; // 购买次数
    
    // 画像版本号（用于乐观锁控制并发更新）
    Long version;
}
```

**画像更新策略**：

| 触发事件 | 更新内容 | 更新方式 |
|---------|---------|---------|
| 用户浏览商品 | 分类权重 +1、价格区间扩展、关键词频次+1、更新浏览时间 | 在线增量（实时） |
| 用户购买商品 | 分类权重 +3（购买比浏览权重更高）、购买次数+1 | 在线增量（实时） |
| 用户登录 | 更新 lastLoginTime、onlineStatus=1 | 在线增量（实时） |
| 每日凌晨 | 全量扫描浏览历史，重新计算所有用户的完整画像 | 离线全量（定时） |

**分类权重归一化**：

```
原始权重：{书籍: 5, 数码: 3, 生活: 1}
归一化后：{书籍: 0.56, 数码: 0.33, 生活: 0.11}

公式：weight_norm(c) = weight(c) / Σ weight(all_categories)
```

#### 4.2.2 商品特征提取

**特征提取流程**：

```
卖家发布商品《算法导论（第4版）》，定价45元，分类=闲置书籍
    ↓
1. 基础特征：categoryId=1, price=45.00
    ↓
2. 中文分词（使用 HanLP 或 Jieba 分词）：
   "算法导论（第4版）" → ["算法", "导论", "第4版"]
    ↓
3. 停用词过滤：
   ["算法", "导论", "第4版"] → ["算法", "导论"]  （"第4版"为停用词）
    ↓
4. 商品特征向量：
   {
     "category_id": 1,
     "category_name": "闲置书籍",
     "price": 45.00,
     "keywords": ["算法", "导论"],
     "seller_id": 123,
     "publish_time": "2026-04-28 16:05:00"
   }
```

#### 4.2.3 匹配度计算

**三维度加权匹配公式**：

```
match_score(user, product) = 
    w1 × category_match(user, product) +    // 分类匹配度，权重 w1=0.5
    w2 × price_match(user, product) +        // 价格匹配度，权重 w2=0.2
    w3 × keyword_match(user, product)        // 关键词匹配度，权重 w3=0.3
```

**各维度计算方式**：

**1. 分类匹配度（权重 0.5）**
```
category_match = user.profile.categoryWeights.get(product.categoryId)
如果用户没有该分类的浏览记录，返回 0
```

**2. 价格匹配度（权重 0.2）**
```
price_match = 
    1.0  （商品价格在用户常浏览的价格区间 [minPrice, maxPrice] 内）
    或
    1 - |product.price - user.avgPrice| / user.avgPrice  （在区间外时，按距离衰减）
    或
    0.5  （用户没有价格数据时，取默认值）
```

**3. 关键词匹配度（权重 0.3）**
```
keyword_match = 
    Σ matched_keyword_count / Σ total_keyword_count
    即：商品关键词与用户关键词的交集大小 / 商品关键词总数
    
例如：商品关键词=["算法", "导论"]，用户关键词=["算法", "Java", "编程"]
交集 = {"算法"}，匹配度 = 1/2 = 0.5
```

#### 4.2.4 两级索引加速

**索引结构**：

| 索引级别 | Key | Value（Redis Set） | 说明 |
|---------|-----|-------------------|------|
| 一级索引（分类-用户） | `idx:category:{categoryId}:users` | 用户ID集合 | 对该分类感兴趣的用户 |
| 二级索引（关键词-用户） | `idx:keyword:{keyword}:users` | 用户ID集合 | 对该关键词感兴趣的用户 |

**索引构建规则**：
- 用户对某个分类的归一化权重 > 0.1 时，加入该分类的索引
- 用户对某个关键词的浏览频次 > 2 时，加入该关键词的索引

**匹配流程优化**：

```
新商品《算法导论》发布（分类=闲置书籍，关键词=["算法", "导论"]）
    ↓
1. 从 Redis 读取 idx:category:1:users → 得到用户集合 U1（约 5000 人）
2. 从 Redis 读取 idx:keyword:算法:users → 得到用户集合 U2（约 200 人）
3. 从 Redis 读取 idx:keyword:导论:users → 得到用户集合 U3（约 50 人）
4. 取 U1 ∩ (U2 ∪ U3) 的交集 → 得到高意向用户集合 U4（约 30 人）
5. 对 U4 中的用户计算完整三维度匹配度
6. 取 TopN（默认 N=10）输出到用户的推荐收件箱
```

**效率对比**：

```
无索引：遍历 100,000 用户 × 计算匹配度 = 100,000 次计算
有索引：分类索引命中 5,000 人 ∩ 关键词索引命中 250 人 = 约 30 人
        只需要计算 30 人的匹配度，效率提升 3000+ 倍
```

### 4.3 数据库设计变更

#### 4.3.1 新增浏览历史表

```sql
CREATE TABLE t_browse_history (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL COMMENT '用户ID',
    product_id INT NOT NULL COMMENT '商品ID',
    browse_duration INT DEFAULT 0 COMMENT '浏览时长（秒）',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '浏览时间',
    FOREIGN KEY (user_id) REFERENCES t_user(id),
    FOREIGN KEY (product_id) REFERENCES t_product(id),
    INDEX idx_user_time (user_id, create_time),
    INDEX idx_user_product (user_id, product_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='浏览历史表';
```

### 4.4 Redis 数据结构设计

| 缓存数据 | 数据结构 | Key 设计 | TTL | 说明 |
|---------|---------|---------|-----|------|
| 用户兴趣画像 | Hash | `user:profile:{userId}` | 24小时 | 增量更新，浏览行为触发刷新 |
| 商品特征 | Hash | `product:feature:{productId}` | 7天 | 发布时写入，下架时删除 |

### 4.5 事件驱动设计

#### 4.5.1 事件类型

| 事件 | 生产者 | 消费者 | 触发动作 |
|------|--------|--------|---------|
| `ProductPublishedEvent` | 商品发布接口 | 匹配引擎 | 提取特征 → 索引查询 → 匹配计算 → 推送收件箱 |
| `ProductBrowsedEvent` | 商品详情页 | 画像更新服务 | 更新分类权重 → 更新关键词 → 更新索引 |
| `ProductPurchasedEvent` | 订单创建接口 | 画像更新服务 | 分类权重 +3 → 更新购买次数 |
| `UserLoginEvent` | 登录接口 | Session 服务 | 更新在线状态 → 更新最后登录时间 |

#### 4.5.2 事件处理流程（以商品发布为例）

```
1. 卖家提交发布商品表单
    ↓
2. ProductController.publish() 调用 ProductService.publish()
    ↓
3. ProductService.publish() 插入数据库
    ↓
4. 发布 ProductPublishedEvent（同步或异步）
    ↓
5. 事件匹配引擎接收事件：
   a. 提取商品特征（含中文分词）
   b. 写入 Redis：product:feature:{productId}
   c. 查询两级索引，获取候选用户集合
   d. 对候选用户计算三维度匹配度
   e. 取 TopN 结果写入 rec:inbox:{userId}
    ↓
6. 用户访问首页/推荐接口时，从收件箱读取推荐结果
```

### 4.6 降级与容错设计

| 故障场景 | 降级级别 | 降级策略 | 用户体验 |
|---------|---------|---------|---------|
| Redis 完全不可用 | L1 | 数据库直查，按分类 + 按浏览量排序推荐 | 推荐从个性化降级为分类热门 |
| 用户画像为空（新用户/冷启动） | L2 | 全局热门商品推荐（按浏览量排序） | 推荐从个性化降级为全局热门 |
| 匹配结果为空（无用户匹配） | L3 | 同分类热门商品推荐 | 推荐从精准匹配降级为同分类热门 |
| 索引数据未就绪 | L4 | 全量用户遍历（限制最多 1000 人） | 匹配范围缩小，但仍有推荐结果 |
| 中文分词服务不可用 | L5 | 仅使用分类和价格维度匹配，跳过关键词维度 | 匹配精度降低，但功能可用 |

**降级链路示意**：
```
正常流程：Redis → 索引查询 → 匹配计算 → 个性化推荐
    ↓ (Redis 不可用)
L1 降级：MySQL → 分类查询 → 按浏览量排序 → 分类热门推荐
    ↓ (画像为空)
L2 降级：MySQL → 热门商品查询 → 按浏览量排序 → 全局热门推荐
```

### 4.8 与分布式课程内容的关联

本方案与分布式软件原理与设计课程的核心内容紧密关联，涉及以下六个关键知识点：

#### 4.8.1 发布-订阅模式（Pub-Sub）

**课程对应知识点**：分布式消息中间件的核心设计模式，Kafka、RabbitMQ 等消息队列的核心机制。

**方案中的体现**：
- **事件模型**：将"新商品上架"建模为事件（Event），商品特征（分类、价格、关键词）作为事件内容
- **订阅规则**：将"用户兴趣画像"建模为订阅（Subscription），用户对哪些分类、价格区间、关键词感兴趣即为订阅条件
- **事件路由**：匹配引擎负责将新商品事件路由到符合条件的用户，相当于消息中间件中的 Broker 路由功能
- **解耦**：卖家和买家之间通过匹配引擎解耦，卖家不需要知道谁会对他的商品感兴趣，买家也不需要主动搜索

**与课程理论的对应关系**：

| 分布式消息中间件概念 | 本方案中的对应 |
|-------------------|--------------|
| Producer（生产者） | 卖家（发布新商品） |
| Event/Message（事件/消息） | 新商品上架事件（含商品特征） |
| Topic（主题） | 商品分类（如"闲置书籍"是一个主题） |
| Subscription（订阅） | 用户兴趣画像（用户订阅了某些分类/关键词） |
| Consumer（消费者） | 买家（接收推荐商品） |
| Broker（路由器） | 匹配引擎（将事件路由到感兴趣的消费者） |
| Routing Key（路由键） | 商品特征向量（分类ID、关键词等） |
| Binding Key（绑定键） | 用户兴趣画像中的偏好维度 |

#### 4.8.2 分布式缓存

**课程对应知识点**：分布式缓存（Redis/Memcached）、缓存一致性、缓存穿透/击穿/雪崩。

**方案中的体现**：
- **缓存数据**：用户兴趣画像（Hash）、商品特征（Hash）、两级索引（Set）、推荐结果（List）、用户 Session 全部存储在 Redis 中
- **多实例一致性**：当系统部署多个 Web 实例时，所有实例从同一个 Redis 集群读取画像和索引数据，保证数据一致性
- **缓存策略**：不同数据设置不同的 TTL，用户浏览行为触发增量更新并刷新 TTL
- **缓存穿透防护**：画像为空时从数据库重建并回填 Redis

#### 4.8.3 分布式索引

**课程对应知识点**：分布式系统中的索引结构（倒排索引、二级索引）、空间换时间思想。

**方案中的体现**：
- 两级索引（分类索引 + 关键词索引）将匹配范围从 10 万用户缩小到约 30 人
- 本质上是倒排索引思想——以内容（分类/关键词）为键，记录包含该内容的用户列表
- 与 Elasticsearch、Solr 等分布式搜索引擎的核心机制一致

#### 4.8.4 离线/在线分离（Lambda 架构）

**课程对应知识点**：Lambda 架构（离线批处理 + 实时流处理）、批流一体。

**方案中的体现**：
- **Batch Layer**：每日凌晨全量计算用户画像，保证准确性
- **Speed Layer**：用户浏览行为触发实时增量更新，保证实时性
- **Serving Layer**：推荐时读取"离线全量 + 在线增量"合并后的数据

#### 4.8.5 分布式锁与并发控制

**课程对应知识点**：分布式锁、乐观锁、并发控制。

**方案中的体现**：
- 使用 Redis 分布式锁防止同一用户画像的并发更新冲突
- 画像数据使用版本号（乐观锁）控制并发写入
- 浏览事件通过队列异步落库，削峰填谷

#### 4.8.6 降级与容错 + 可观测性

**课程对应知识点**：分布式系统的容错设计、服务降级、可观测性（Metrics/Tracing/Logging）。

**方案中的体现**：
- 五级降级策略，保证依赖组件故障时功能不中断
- Metrics/Tracing/Logging 三大支柱全面监控系统运行状态

## 七、风险分析

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|---------|
| 用户兴趣画像不准确 | 推荐效果差 | 中 | ① 逐步优化权重计算 ② 引入更多行为信号（浏览时长、收藏、分享） ③ 引入用户反馈机制（点赞/不感兴趣） |
| 中文分词效果差 | 关键词匹配不准确 | 中 | ① 使用成熟的分词库（HanLP） ② 建立自定义词典（校园二手商品专用词） ③ 分词不可用时降级为仅使用分类和价格维度 |
| Redis 引入后运维复杂度 | 增加系统维护成本 | 低 | ① 使用 Docker 部署 Redis ② 配置 Redis 持久化（RDB + AOF） ③ 搭建 Redis Sentinel 高可用 |
| 新用户冷启动 | 新用户无个性化推荐 | 高 | ① 降级为热门商品推荐 ② 引导用户选择兴趣分类 ③ 积累少量浏览行为后逐步个性化 |
| 高并发下 Redis 压力 | 缓存响应变慢 | 低 | ① 配置 Redis 连接池 ② 热点数据设置合理的 TTL ③ 考虑 Redis 集群分片 |
| 多实例并发更新画像 | 画像数据不一致 | 中 | ① Redis 分布式锁 ② 版本号乐观锁 ③ 最终一致性设计 |
