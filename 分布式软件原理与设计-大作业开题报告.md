# 分布式软件原理与设计大作业开题报告

## 一、课题名称
校园二手交易平台的高并发优化、分布式演进与服务治理能力建设

## 二、研究背景与意义
校园二手交易平台在毕业季、开学季等时间段容易出现访问与交易请求的集中爆发。现有系统虽已实现完整业务闭环，但在真实线上环境中会面临三类关键挑战：

- **高并发正确性**：抢同一件商品、重复点击下单、热点商品详情频繁访问等场景下，必须保证一致性与正确性，避免超卖与重复订单。
- **分布式可扩展性**：从单机部署扩展到多实例部署时，需要解决会话、缓存、计数、文件存储等状态耦合导致的不一致问题，支持水平扩容。
- **服务治理与可观测性**：面对突发流量与局部故障，需要限流、降级、熔断等保护能力，同时具备监控与追踪手段以快速定位问题。

本课题以现有 SSM（Spring + Spring MVC + MyBatis + JSP）校园二手交易平台为基础，围绕课程的“高并发原理、分布式设计、服务治理”内容，提出可落地的工程化优化与验证方案，实现从“可运行”向“可扩展、可治理”的演进。

## 三、现有系统概述

### 3.1 已实现的主要功能
- **用户体系**：注册、登录（含验证码校验）、退出、个人中心信息维护；权限拦截（普通用户/管理员）。
- **商品体系**：商品发布（含图片上传）、编辑、上下架、删除；商品列表展示、分页、搜索、分类筛选；商品详情页。
- **交易体系**：立即购买生成订单；买家查看“我的订单”；卖家查看“我的销售”；卖家确认完成/取消订单（取消后恢复商品在售）。
- **后台管理**：用户/商品/订单管理与简单统计展示（用户数、订单数、总交易额等）。
- **推荐能力**：相似商品推荐（同分类）与个性化推荐（基于浏览历史的简化内容推荐）。

### 3.2 现存问题（与课程内容对应）
1. **高并发问题**：下单流程存在并发竞态窗口，缺少幂等保护；热点访问对数据库读写放大；部分逻辑存在 N+1 查询导致性能下降。
2. **分布式设计问题**：会话、本地缓存与文件存储依赖单机；一旦多实例部署，容易出现登录态丢失、数据不一致与图片不可访问等问题。
3. **服务治理问题**：缺少限流/熔断/降级策略与可观测能力；异常返回风格不统一，联调与故障统计困难。

## 四、研究目标与预期效果（可量化指标）

### 4.1 高并发正确性目标
- **避免超卖**：同一商品在高并发购买场景下，最终有效订单数量符合业务规则（典型为 1 单成功），其他请求得到明确失败响应（如“已售出”）。
- **下单幂等**：同一业务请求（同一幂等标识 `requestId`）重复提交不会生成多笔订单。
- **稳定性提升**：下单与热点接口在并发提升时，错误率可控，延迟不会出现非线性雪崩。

### 4.2 分布式可扩展目标
- **多实例一致性**：双实例或多实例部署后，用户登录态保持一致；浏览历史/推荐依据一致；重启不丢关键状态。
- **无状态化部署**：应用实例可水平扩容，新增节点不引入明显的数据分裂或功能异常。

### 4.3 服务治理与可观测目标
- **流量保护有效**：热点接口具备限流能力；下游异常时具备降级/熔断策略，避免级联故障。
- **可观测性可用**：具备接口耗时、错误率、吞吐、数据库连接池与慢查询等指标展示；能够支持定位性能瓶颈。
- **响应与错误码统一**：统一错误码与响应结构，便于前端处理、统计分析与告警。

## 五、研究内容与技术方案

### 5.1 高并发优化：关键业务原子化与幂等化

#### 5.1.1 下单流程的并发竞态修复（原子化）
现有流程存在“先查在售、再插订单、再更新商品状态”的竞态窗口。改造方案：

- 将商品状态更新改为**条件更新（Compare-And-Set）**：
  - `UPDATE t_product SET status = 1 WHERE id = ? AND status = 0`
  - 通过影响行数判断是否抢购成功，实现数据库层原子性，消除竞态窗口。
- 订单创建与状态更新保持事务边界清晰，必要时调整为“先抢占商品（条件更新成功）→ 再创建订单”，失败即回滚或返回失败。

#### 5.1.2 下单幂等（防重复提交）
- 引入客户端生成的唯一幂等标识 `requestId`（一次购买请求唯一）。
- 数据库层添加唯一约束（订单表新增 `request_id` 字段并建唯一索引，或新增幂等记录表）。
- 服务端逻辑：收到重复 `requestId` 时返回同一订单结果（或明确提示“请求已处理”），避免重复插入订单。

#### 5.1.3 热点读写与查询优化
围绕“热点商品详情”“推荐查询”两类压力点：

- **浏览量统计**：将同步写库改为缓存计数（如 Redis `INCR`）+ 异步/批量落库，降低写放大，平滑高峰。
- **推荐查询优化**：避免 N+1 查询（循环 `findById`），改为批量查询或缓存化关键数据，降低数据库压力并提升响应速度。

#### 5.1.4 验证方案（压测与正确性验证）
- 并发下单同一商品：验证“最多 1 单成功”，其余请求失败响应一致。
- 重复提交同一 `requestId`：验证仅生成 1 笔订单，重复请求返回一致结果。
- 热点商品详情并发访问：对比改造前后数据库写入量与接口 P95/P99 延迟变化。

### 5.2 分布式演进：状态外置与无状态化

#### 5.2.1 会话外置（多实例登录一致）
当前使用本地 `HttpSession`，多实例下会出现“偶发掉线”。改造方案：

- 引入 **Redis + Session 共享**（如 Spring Session 或等价方案）将会话状态外置。
- 目标：在负载均衡轮询或节点切换时，用户仍保持登录态。

#### 5.2.2 浏览历史外置（推荐依据一致）
当前浏览历史存于进程内存，重启丢失且多实例不一致。改造方案：

- 将浏览历史迁移至 Redis（List/SortedSet），并实现 LRU 或容量控制策略。
- 目标：跨实例一致、服务重启不丢失、推荐结果更稳定。

#### 5.2.3 计数与缓存外置
- 浏览量计数使用 Redis 作为写入缓冲，定时/异步落库，数据库仅承担最终一致的持久化。
- 目标：高峰期数据库写压力显著降低。

#### 5.2.4 文件存储解耦
当前图片存储依赖应用服务器本地磁盘，多实例下会出现裂图。改造方案：

- 迁移图片到对象存储（可用 MinIO/OSS/COS 等），数据库只存 URL。
- 目标：任何实例均可访问图片，节点扩缩容不影响资源可用性。

### 5.3 服务治理：限流、降级、熔断与可观测性

#### 5.3.1 流量治理（限流/防刷）
对高风险接口（登录、下单、详情、推荐）建立流量保护：

- 在统一入口（网关或过滤器层）按 IP/用户维度限流。
- 超限时给出明确响应（如“系统繁忙，请稍后再试”），保护后端与数据库。

#### 5.3.2 降级与熔断
对非核心或可替代能力设置降级：

- 推荐不可用时回退为热门商品推荐。
- 缓存不可用时返回安全默认值并控制回源流量，避免雪崩。

#### 5.3.3 可观测性建设
构建最小可用的观测体系：

- 指标：QPS、成功率、错误率、P95/P99 延迟、连接池指标、慢查询统计等。
- 展示：监控看板与基础告警阈值（如错误率突增、延迟突增、连接池耗尽风险）。
- 追踪：对关键链路（下单）提供请求链路定位能力（可按课程要求选型）。

#### 5.3.4 统一错误码与响应结构
- 统一对外返回结构（如 `Result<T>`），规范错误码与错误消息。
- 消除接口返回风格不一致带来的联调成本与故障统计困难。

## 六、技术路线与架构演进计划

### 阶段 A：并发正确性优先（先对齐“正确”）
- 下单条件更新实现原子化，消除竞态窗口
- 幂等机制落地（`requestId` + 唯一约束）
- 热点写优化（浏览量计数缓存化）
- 压测验证正确性与稳定性（并发下单、重复提交、热点详情）

### 阶段 B：状态外置（实现“可扩”）
- Redis 外置会话，实现多实例一致登录
- 浏览历史/推荐依据迁移 Redis
- 计数异步落库
- 图片迁移到对象存储，解耦本地磁盘
- 多实例部署验证（轮询访问、重启验证、数据一致性验证）

### 阶段 C：治理与观测（实现“可管”）
- 接口限流策略与降级回退落地
- 可观测体系（指标/看板/告警）与关键链路追踪
- 统一错误码与响应结构，补齐故障演练与对比报告

## 九、风险分析与应对
- **风险 1：并发修复不彻底导致仍出现少量重复订单**
  - 应对：以数据库条件更新与唯一约束为准，补齐压测回归用例与边界条件验证。
- **风险 2：缓存引入后一致性策略不清导致业务混乱**
  - 应对：明确一致性分级：下单等关键链路追求强一致；浏览量/历史记录采用最终一致。
- **风险 3：治理与可观测建设成本较高**
  - 应对：先完成最小集合（限流 + 关键指标 + 统一错误码），再逐步增强链路追踪与告警体系。
