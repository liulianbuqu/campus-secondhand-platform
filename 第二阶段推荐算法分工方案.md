# 第二阶段推荐算法分工方案（5人版）

---

## 总体策略

**核心算法**：基于内容的推荐（Content-Based Filtering）

**为什么选 CB 而不是协同过滤？**
1. 校园二手平台用户量和商品量有限，协同过滤的"群体智慧"优势发挥不出来
2. 协同过滤有严重的新商品冷启动问题（新商品没人浏览过就无法推荐）
3. CB 算法与你们的"发布-订阅"架构天然匹配——商品特征作为事件内容，用户画像作为订阅条件
4. 你们已有浏览历史、商品分类等数据基础，CB 可以直接用

**两阶段策略**：

```
第二阶段整体
├── 子阶段①：CB算法基础实现（2人） ← 先跑通核心流程
│   ├── 成员A：用户兴趣画像系统
│   └── 成员B：商品特征提取 + 匹配引擎
│
└── 子阶段②：算法优化（3人） ← 在基础上做优化
    ├── 成员C：两级索引 + 推荐收件箱
    ├── 成员D：降级容错 + 监控看板
    └── 成员E：压测对比 + 演示材料
```

## 子阶段①：CB算法基础实现（2人）

**目标**：跑通"发布商品 → 提取特征 → 匹配用户 → 推送收件箱"的完整链路

### 成员A：用户兴趣画像系统

**一句话**：给每个用户建"兴趣档案"，记录他喜欢什么分类、什么价位的商品

| 任务 | 具体内容 | 涉及文件 | 验收方法 |
|------|---------|---------|---------|
| ① 定义画像数据结构 | 分类权重Map、价格区间(min/max/avg)、关键词频次Map、版本号 | `UserProfile.java`（新增实体） | 文档清晰定义 ✅ |
| ② 画像增量更新 | 用户浏览商品 → 分类权重+1、更新价格区间、关键词频次+1 | `UserProfileService.java`（新增） | 浏览3本计算机书→Redis里"计算机"权重最高 ✅ |
| ③ 画像持久化 | 画像数据存Redis（Hash）+ MySQL（JSON）双写 | `UserProfileMapper.java/.xml`（新增） | 重启后画像不丢失 ✅ |
| ④ 画像可视化接口 | 写一个接口返回当前用户的画像JSON | `UserProfileController.java`（新增） | 访问接口能看到完整画像数据 ✅ |

**核心代码示例（画像数据结构）**：
```java
// Redis Hash 存储结构
// Key: user:profile:{userId}
// Field: "categoryWeights" → {"1": 5, "3": 3, "5": 1}
// Field: "priceRange" → {"min": 10.0, "max": 99.0, "avg": 45.0}
// Field: "keywords" → {"算法": 3, "Java": 2, "编程": 1}
// Field: "version" → 3
```

**依赖关系**：依赖第一阶段成员1的Redis配置（已就绪）

### 成员B：商品特征提取 + 匹配引擎

**一句话**：新商品发布后，提取它的特征，然后找到哪些用户可能感兴趣

| 任务 | 具体内容 | 涉及文件 | 验收方法 |
|------|---------|---------|---------|
| ① 中文分词 | 商品标题分词："算法导论第4版"→["算法","导论","第4版"]→过滤停用词→["算法","导论"] | `ProductFeatureService.java`（新增） | 控制台打印分词结果 ✅ |
| ② 商品特征提取 | 提取分类ID、价格、关键词，写入Redis Hash | `ProductFeatureService.java` | Redis里能查到商品特征 ✅ |
| ③ 匹配度计算 | 三维度加权：分类0.5 + 价格0.2 + 关键词0.3 | `MatchService.java`（新增） | 发布《算法导论》，计算机系用户匹配度0.85 ✅ |
| ④ 事件匹配引擎 | 商品发布事件触发 → 提取特征 → 匹配用户 → 推送到收件箱 | `MatchEngine.java`（新增） | 发布后1秒内目标用户收到推荐 ✅ |

**匹配度计算公式**：
```
match_score = 0.5 × category_match + 0.2 × price_match + 0.3 × keyword_match

category_match = 用户画像中该分类的归一化权重
price_match   = 商品价格在用户价格区间内→1.0，否则按距离衰减
keyword_match = 商品关键词与用户关键词的交集大小 / 商品关键词总数
```

**依赖关系**：
- 依赖成员A的画像数据
- 依赖第一阶段成员3的商品发布事件（已就绪）

## 子阶段②：算法优化（3人）

**目标**：在基础CB算法上做性能优化、容错保障和效果验证

### 成员C：两级索引 + 推荐收件箱

**一句话**：建索引加速匹配，别每次都遍历所有人

| 任务 | 具体内容 | 涉及文件 | 验收方法 |
|------|---------|---------|---------|
| ① 分类-用户索引 | 对某分类感兴趣的用户ID集合（Set） | `IndexService.java`（新增） | Redis里 `idx:category:1:users` 有数据 ✅ |
| ② 关键词-用户索引 | 对某关键词感兴趣的用户ID集合（Set） | `IndexService.java` | Redis里 `idx:keyword:算法:users` 有数据 ✅ |
| ③ 索引查询优化 | 匹配时先查索引缩小范围，再计算匹配度 | `MatchEngine.java`（改造） | 日志打印"索引命中30人，无需遍历10万人" ✅ |
| ④ 推荐收件箱完善 | 用户读取收件箱，按匹配度排序展示 | `InboxService.java`（新增） | 推荐列表按匹配度从高到低展示 ✅ |

**索引加速原理**：
```
无索引：遍历 100,000 用户 × 计算匹配度 = 100,000 次计算
有索引：分类索引命中 5,000 人 ∩ 关键词索引命中 250 人 = 约 30 人
        只需要计算 30 人的匹配度，效率提升 3000+ 倍
```

**依赖关系**：依赖成员A的画像数据 + 成员B的匹配引擎

---

### 成员D：降级容错 + 监控看板

**一句话**：Redis挂了推荐也不能挂，还要能看到系统状态

| 任务 | 具体内容 | 涉及文件 | 验收方法 |
|------|---------|---------|---------|
| ① L1降级：Redis不可用 | Redis挂了→自动查数据库，按分类+浏览量推荐 | `DegradeService.java`（新增） | 停掉Redis，推荐依然有结果 ✅ |
| ② L2降级：新用户冷启动 | 新用户没画像→推荐热门商品 | `DegradeService.java` | 新注册用户看到热门推荐 ✅ |
| ③ 指标采集 | 统计匹配耗时、推送量、缓存命中率 | `MetricsService.java`（新增） | 控制台每10秒打印一次指标 ✅ |
| ④ 监控页面 | 简单的HTML页面展示实时指标 | `monitor.html`（新增） | 访问 /monitor.html 看到数据 ✅ |

**降级链路**：
```
正常流程：Redis → 索引查询 → 匹配计算 → 个性化推荐
    ↓ (Redis 不可用)
L1 降级：MySQL → 分类查询 → 按浏览量排序 → 分类热门推荐
    ↓ (画像为空)
L2 降级：MySQL → 热门商品查询 → 按浏览量排序 → 全局热门推荐
```

**依赖关系**：依赖成员A/B/C的所有功能（降级是在它们基础上的兜底）
